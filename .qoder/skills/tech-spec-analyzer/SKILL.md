# Tech Spec 分析 Skill

分析技术规格书的完整性、准确性和可实现性，确保代码生成前规格质量达标。

---

## 触发条件

- 用户指令："分析 tech-spec"、"检查规格书"、"评估规格完整性"
- tech-spec.yml 生成完成后，代码生成前
- 用户质疑规格书是否足够详细以支持开发

---

## 输入

| 输入项 | 类型 | 说明 |
|--------|------|------|
| tech_spec | file | 技术规格书路径（tech-spec.md，Markdown 格式） |
| feature_definition | object | Feature 原始定义（可选，用于对比） |
| target_services | array | 涉及的服务列表 |
| code_generation_standards | object | 代码生成规范（包含 12-spec-generation-constraints.md） |

---

## 分析维度

### 1. 完整性检查

检查 tech-spec.md 是否包含所有必需章节：

```
□ 基本信息（Feature ID、名称、描述）
□ 服务分层定义（facade/application/data）
□ 接口定义（内部接口 + 门面接口）
□ 数据模型（表结构、字段、索引）
□ 业务规则（校验规则、状态流转）
□ 调用时序图（查询/写操作/跨服务依赖，Mermaid 格式）
□ 业务流程图（Mermaid 格式，如需要）
□ 实现计划（分层顺序、依赖关系）
□ 测试数据（初始化数据、测试场景）
```

### 2. 接口定义检查

| 检查项 | 通过标准 | 常见问题 |
|--------|----------|----------|
| 路径规范 | 符合 `/admin/api/v1/` 或 `/inner/api/v1/` 前缀 | 路径前缀错误、版本号缺失 |
| HTTP 方法 | GET/POST/PUT/DELETE 使用合理 | 查询用 POST、更新用 GET 等错误 |
| 参数定义 | 每个参数有名称、类型、必填、说明 | 参数类型缺失、必填属性未标注 |
| 响应定义 | 明确响应类型和结构 | 响应类型为 "Object" 等模糊定义 |
| 字段映射 | 请求/响应字段与数据模型一致 | 字段名不一致、类型不匹配 |

### 3. 数据模型检查

| 检查项 | 通过标准 | 常见问题 |
|--------|----------|----------|
| 表名规范 | 符合 `aim_xxx` 命名规范 | 表名缺少前缀、使用驼峰命名 |
| 字段完整 | 每个字段有名称、类型、约束、说明 | 字段类型缺失、长度未定义 |
| 主键定义 | 明确主键策略（自增/雪花ID） | 主键策略未指定 |
| 索引设计 | 查询字段有索引、唯一约束明确 | 高频查询字段无索引 |
| 软删除 | 包含 `is_deleted` 或 `deleted_at` | 删除策略未定义 |
| 审计字段 | 包含 `create_time`、`update_time` | 缺少创建/更新时间字段 |

### 4. 业务规则检查

| 检查项 | 通过标准 | 常见问题 |
|--------|----------|----------|
| 校验规则 | 明确字段校验规则（长度、范围、格式） | 校验规则缺失或模糊 |
| 唯一性约束 | 明确唯一性约束（单字段/联合唯一） | 唯一性约束未定义 |
| 状态流转 | 有状态定义和流转图/表 | 状态值未枚举、流转不完整 |
| 业务逻辑 | 复杂逻辑有伪代码或流程描述 | 复杂逻辑仅一句话描述 |
| 异常场景 | 定义错误码和异常处理 | 异常场景未考虑 |

### 5. 时序图和流程图检查

| 检查项 | 通过标准 | 常见问题 |
|--------|----------|----------|
| 查询时序图 | 展示完整调用链（门面→Feign→应用→DB） | 缺少 Feign 调用环节 |
| 写操作时序图 | 包含 operatorId 解析和传递 | 未标注事务边界 |
| 跨服务时序图 | 展示服务间调用关系 | 跨服务调用未体现 |
| 业务流程图 | 展示业务处理流程 | 流程不完整 |
| Mermaid 语法 | 语法正确、可渲染 | 语法错误、无法显示 |
| 参与者定义 | 所有服务/组件都有对应参与者 | 参与者缺失 |

### 6. 规范合规性检查

基于 `12-spec-generation-constraints.md` 检查规格是否符合代码生成约束：

| 检查项 | 通过标准 | 常见问题 |
|--------|----------|----------|
| Controller 命名 | 符合 AdminController/InnerController 规范 | 命名不符合规范 |
| 路径前缀 | /admin/api/v1/ 或 /inner/api/v1/ | 路径前缀错误 |
| 参数校验 | @Valid（门面）或手动校验（内部） | 校验方式错误 |
| DTO 命名 | Request/Response 或 ApiRequest/ApiResponse | 命名不符合分层规范 |
| Service 分层 | Query/Manage/Application 职责清晰 | 分层混乱 |
| DO 实体 | Aim{Name}DO 格式，字段完整 | 命名或字段不符合规范 |
| 数据库表 | aim_{模块}_{业务名} 格式 | 表名不符合规范 |

### 7. 可实现性评估

| 评估项 | 评估标准 | 风险等级 |
|--------|----------|----------|
| 技术可行性 | 现有技术栈可支持 | 高：需要引入新技术 |
| 依赖清晰度 | 外部依赖明确且可用 | 高：依赖未确认或不存在 |
| 复杂度适中 | 单 Feature 复杂度可控 | 高：需要拆分为多个 Feature |
| 数据量评估 | 预估数据量合理 | 中：数据量预估缺失 |
| 性能要求 | 响应时间、并发量明确 | 中：性能要求未定义 |

## 分析流程

### 步骤 1：读取 tech-spec

1. 读取 tech-spec.yml 文件
2. 解析 YAML 结构
3. 提取各章节内容

### 步骤 2：执行检查清单

按以下顺序执行检查：

1. **完整性检查** → 识别缺失章节
2. **接口定义检查** → 识别接口问题
3. **数据模型检查** → 识别模型问题
4. **业务规则检查** → 识别规则缺陷
5. **时序图和流程图检查** → 识别调用链和业务流程问题
6. **规范合规性检查** → 识别是否符合代码生成约束（基于 12-spec-generation-constraints.md）
7. **可实现性评估** → 识别实施风险

### 步骤 3：汇总问题

按严重程度分类：

| 级别 | 说明 | 处理方式 |
|------|------|----------|
| **阻塞** | 必须修复才能进入代码生成 | 标记为 blocked，返回修复清单 |
| **警告** | 建议修复，但不阻塞流程 | 列出建议，用户决定是否修复 |
| **提示** | 可选优化，不影响功能 | 列出优化建议 |

### 步骤 4：生成分析报告

输出格式：

```markdown
# Tech Spec 分析报告

## 基本信息

- Feature ID: {F-XXX}
- Feature 名称: {功能名称}
- 分析时间: {YYYY-MM-DD}
- 分析结果: 通过 / 有条件通过 / 不通过

## 检查摘要

| 检查维度 | 状态 | 问题数 |
|----------|------|--------|
| 完整性 | ✅ 通过 | 0 |
| 接口定义 | ⚠️ 警告 | 2 |
| 数据模型 | ✅ 通过 | 0 |
| 业务规则 | ❌ 阻塞 | 1 |
| 时序图/流程图 | ✅ 通过 | 0 |
| 规范合规性 | ⚠️ 警告 | 1 |
| 可实现性 | ✅ 通过 | 0 |

## 阻塞问题（必须修复）

### 问题 1: 状态流转未定义

**位置**: business_rules.state_machine
**描述**: {业务实体}状态（ACTIVE/INACTIVE）的流转规则未定义
**影响**: 无法确定状态变更的业务逻辑
**建议**: 补充状态流转图或状态流转表

## 警告问题（建议修复）

### 问题 1: 分页查询默认页大小未指定

**位置**: interfaces.facade[2].parameters.pageSize
**描述**: 分页查询接口未指定默认页大小
**建议**: 明确 default = 10

## 优化建议

1. 建议为表添加 `sort_order` 字段支持排序
2. 建议补充接口的异常响应示例

## 结论

**状态**: 有条件通过

存在 1 个阻塞问题，修复后可进入代码生成阶段。
```

  - 数据模型：✅ 通过 / ⚠️ 警告 / ❌ 不通过
  - 业务规则：✅ 通过 / ⚠️ 警告 / ❌ 不通过

## 常见问题模式

### 模式 1：接口定义不完整

**症状**: 接口缺少请求/响应定义
**处理**: 标记阻塞，要求补充 DTO 定义

### 模式 2：数据模型与接口不一致

**症状**: 接口字段名与数据库字段名不一致
**处理**: 标记警告，建议统一命名

### 模式 3：业务规则过于简单

**症状**: 复杂业务仅一句话描述
**处理**: 标记警告，建议补充伪代码或流程图

### 模式 4：依赖未明确

**症状**: 依赖其他服务但未定义 Feign 接口
**处理**: 标记阻塞，要求补充依赖接口定义

### 模式 5：时序图缺失或不完整

**症状**: 缺少时序图或时序图未展示完整调用链
**处理**: 标记警告，建议补充标准时序图模板
