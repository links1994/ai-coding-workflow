# Rule 创建指南

> 本文档指导如何在流程设计中创建或配置 Rules。

---

## 何时使用

在流程设计过程中，当需要：
- 为项目定义编码规范或约束条件
- 设置特定文件类型的处理规则
- 配置模型自动触发的场景规则
- 建立项目级的强制执行标准

---

## 创建流程

### 步骤1：分析约束需求

确定需要规则约束的方面：
- 代码风格（命名规范、格式化要求）
- 架构约束（分层规则、依赖限制）
- 文档规范（注释格式、文档结构）
- 流程规范（提交信息、审查要求）

### 步骤2：选择规则类型

| 类型 | 适用场景 | 示例 |
|------|----------|------|
| **始终生效** | 项目级强制标准 | 所有代码必须使用 UTF-8 编码 |
| **指定文件生效** | 特定文件类型约束 | *.java 文件必须包含类注释 |
| **模型决策** | 场景化自动触发 | 生成单元测试时触发测试规范 |
| **手动引入** | 按需工作流 | 代码审查时手动引入审查清单 |

### 步骤3：设计规则内容

规则文件结构：

```markdown
# [规则名称]

## 适用范围

[描述规则适用的场景和文件类型]

## 约束规则

### 必须遵守（MUST）
- 规则1
- 规则2

### 应该遵守（SHOULD）
- 规则1
- 规则2

### 禁止事项（MUST NOT）
- 禁止1
- 禁止2

## 示例

### 正确示例
```java
// 符合规范的代码示例
```

### 错误示例
```java
// 不符合规范的代码示例
```

## 参考
- [相关文档链接]
```

---

## 设计原则

**必须遵守：**
1. 规则总字符数不超过 100,000（约 100KB）
2. 使用清晰的 Markdown 格式
3. 包含具体的代码示例
4. 按重要性分级（MUST/SHOULD/MUST NOT）

**应该遵守：**
1. 规则描述简洁明确，避免歧义
2. 与 AGENTS.md 保持一致
3. 根据实际使用效果迭代优化
4. 优先使用"始终生效"处理核心标准

**禁止事项：**
1. 不在规则中插入图片或超链接
2. 不创建重复或冲突的规则
3. 不设置过于复杂难以执行的规则

---

## 输出格式

为流程设计提供：
1. **规则清单**：列出需要创建的规则
2. **规则类型**：为每条规则指定类型
3. **规则文件内容**：完整的规则文件 Markdown
4. **文件路径**：`.qoder/rules/[rule-name].md`

---

## 快速参考

**规则文件命名规范**
- 使用小写字母和连字符
- 示例：`coding-style.md`, `java-conventions.md`, `api-design.md`

**常见规则类型选择**
- 代码风格 → 始终生效
- 语言特定规范 → 指定文件生效（如 `*.java`）
- 测试规范 → 模型决策（场景：生成单元测试）
- 审查清单 → 手动引入
